## Context

이 변경은 런타임 학습 기능이 아니라 배포/운영 보조다. 핵심 하네스 경로를 건드리지 않고, 세션 시작 시 경량 버전 체크만 제공하는 설계가 적합하다.

## Goals / Non-Goals

**Goals:**
- auto-update-checker를 default-off로 둔다.
- 체크 실패가 세션 시작이나 기존 경로를 방해하지 않게 한다.
- 알림은 warn-only와 쿨다운으로 제한한다.

**Non-Goals:**
- 자동 업데이트를 수행하지 않는다.
- 릴리스 운영 기능을 core harness loop에 결합하지 않는다.
- 외부 트렌드 자동 수집, todo-continuation, autopilot을 포함하지 않는다.

## Decisions

### D1: 세션 시작 시 비차단 체크만 수행한다

버전 확인은 session.start 계열 훅에서 실행하되, 실패하면 그대로 통과한다. 대안인 blocking check는 네트워크 문제를 사용자 세션에 전파한다.

### D2: 알림은 warn-only로 둔다

업데이트가 있더라도 자동 실행하지 않고 사용자에게만 알린다. 대안인 auto-install은 사용자가 원하지 않을 수 있고 위험하다.

### D3: 쿨다운은 로컬 상태로 관리한다

알림 빈도는 마지막 체크/알림 시점을 파일에 기록해 제어한다. 대안인 매 세션 무조건 알림은 피로도를 높인다.

## Risks / Trade-offs

- [매 세션 네트워크 요청 부담] → default-off와 쿨다운으로 줄인다.
- [버전 체크 실패] → 실패는 무시하고 경고만 남긴다.
- [알림 피로도] → TTL과 쿨다운으로 제한한다.

## Migration Plan

1. default-off 상태로 배포한다.
2. 선택적으로 enable 한 환경에서만 체크를 관찰한다.
3. 알림 빈도와 실패율을 본 뒤 TTL / 쿨다운 값을 조정한다.
4. 문제가 생기면 체크 기능만 끄고 core harness는 그대로 둔다.

## Open Questions

- 알림 TTL을 세션 기준으로 둘지 시간 기준으로 둘지
- 쿨다운 기록을 전역으로 둘지 프로젝트별로 둘지
